Repozytoria Git

System kotnroli werjsi - jest to stystem pozwalający na sprawdzanie zmian w kodzie źródłowym projektu oraz pomaga w pracy zespołowej.
Aby zacząć korzystać z GIT-a wystarczy pobrać Git brush. Można utworzyć projekt komendą lub otwrzyć folder za pomogcą Git Brush.Wystarczy otworzyc folder za pomocą git bush i wpisać 'git init' i repozytorium zostanie utworzone.
Aby zaoszczędzić czas oraz miejsec w zdalnym repozytorium towrzymi plik o nazwie '.gitignore'. Posiada on w sobie rozszerzenia plików, których ma nie zapisywać przez co na zdalnym i lokalnym repozytorium jest to co niezbędne.
Pliki mają kilka stanów: Untracked, Modified, Commited. Untracked to stan pliku, w którym git nie śledzi zmian, które się w nim dzieją. Aby przejść do następnego stanu(Commited) wystarczy użyć komendy 'git add (nazwa pliku lub '.' aby dodać wszystkie pliki)'. Modified jest to stan gdzie git śledzi zmiany w tym pliku. Tu any przejść do nastepnego stanu wystarczy zrobić tak samo jak byśmy dodawali nowy plik. Następnym stanem jest Commited czy plik, którego zmiany zostały zapisane na lokalnym repozytorium i może zostać wysłany na zdalne repozytorium. Oczywiście można przestać śledzić plik - aby go usunąć - lub go wyciągnąć go ze stanu Commited za pomocą komendy 'git restore'.
Git może tworzyć gałęzie(branch) jest to odbicie od głównej ścieżki rozwoju aby przetesować inną funkcje nie naruszając głównej strukyury programu. Nową gałąź tworzymy za pomocą komendy 'git branch (nazwa gałęzi), to polecenie tylko tworzy i nie przenosi na nowa gałąź. Oczywiście jest możliwość przełączania się pomiędzy gałęziami inaczej nie miało by to sensu. Służy do tego komenda 'git checkout (nazwa gałęźi)'. Jednak skąd Git wie na której gałęźi jesteśmy? Utrzymuje on znacznik HEAD nad branchem, w którym pracujemy.
Same gałęzie można połączyć ze sobą za pomocą 'git merge (nazwa gałęzi)'. Git merge szuka commita bazowego spośród końcówek gałęźi, które łączymy. Gdy go znajdzie to utworzy 'commit scalania'. Jednak jeżeli znajdzie konflikt to użytkownik musi zareagować.
Można podejżeć historię repozytorium za pomocą komendy 'git log'. Wyświetla ona wszystkie zatwierdzone zmiany w odwrotnej kolejności czyli najnowsze na samej górze. Jest wiele możliwości wyświetlenia logu. Można użyć dodatkowego polecenia --graph i zostanie zbudowany ładny graf w ASCII
. Istnieją też dalne repozytoria takie jak GitHub. Zdalne repozytorium to wersja twojego projektu utrzymywana na serwerze dostępnym poprzez Internet lub inną sieć. Możesz mieć ich kilka, z których każde może być tylko do odczytu lub zarówno odczytu jak i zapisu. Wykorzystywana jest komenda 'git push' aby wysłać tam swój projekt.
Ale aby wysłać trzeba najpierw dodać połączyć swoje konto np. na GitHub z gitem loklanym. Można również pobierać swoje repozytorai za pomocą komendy 'git pull origin main'. Ale napierw zalecane jest użycie komendy 'git fetch', która pobiera zdalne repozytorium nienaruszająć lokalnego zaś 'git pull' odrazu zamienia pliki.
